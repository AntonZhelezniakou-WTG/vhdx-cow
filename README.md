# vhdx-cow

VHDX-backed Copy-on-Write for git worktrees on Windows.

Build artifacts (e.g. `bin/`) in the main repository live on a parent VHDX disk. Each git worktree gets a _differencing_ (child) VHDX that stores only its own changes — unmodified blocks are read transparently from the immutable parent. Resetting a worktree to the parent state takes milliseconds: the child disk is deleted and recreated from scratch.

Mounting a VHDX to an NTFS folder requires administrator privileges. A Windows Service handles all privileged disk operations; a user-space CLI communicates with it over a local gRPC/named-pipe channel.

---

## Architecture

```
main repo
└── bin/  ←── main-overlay.vhdx  (differencing)
                      │
                      ▼
              main.vhdx  (parent, READ-ONLY)
                      │
          ┌───────────┴───────────┐
          ▼                       ▼
  wt1-child.vhdx          wt2-child.vhdx
  worktree1/bin/          worktree2/bin/
```

The parent VHDX is **immutable**. Both the main repository overlay and each worktree child reference it as a read-only base. Modifying the parent (via the `publish` command) detaches all children, merges the overlay into the parent, and recreates all children from the new baseline.

### Components

| Project | Role |
|---------|------|
| `VhdxCow.Contracts` | Protobuf service definition, shared constants (`ServiceConstants`) |
| `VhdxCow.Service` | Windows Service — gRPC server over named pipe, all privileged VHDX operations |
| `VhdxCow.Client` | Client library — typed async wrapper over the gRPC channel |
| `VhdxCow.Cli` | Command-line tool for end users |

### IPC

The CLI connects to the service via a Windows named pipe (`\\.\pipe\VhdxCowService`) using gRPC/HTTP2. Named pipe ACLs grant access to `BUILTIN\Users`; `TokenImpersonationLevel.Anonymous` prevents privilege escalation from the client side.

### VHDX operations

The service calls the Win32 VirtDisk API directly via P/Invoke (generated by [CsWin32](https://github.com/microsoft/CsWin32)) — no dependency on Hyper-V PowerShell cmdlets or the Hyper-V role.

Key API calls: `CreateVirtualDisk` · `OpenVirtualDisk` · `AttachVirtualDisk` · `DetachVirtualDisk` · `GetVirtualDiskPhysicalPath` · `MergeVirtualDisk` · `SetVolumeMountPoint` · `DeleteVolumeMountPoint`

Attached disks use the `PERMANENT_LIFETIME` flag — mounts survive service restarts. Active mounts are tracked in `%ProgramData%\VhdxCow\state.json` and reconciled on startup.

## Requirements

- Windows 10/11 or Windows Server 2019+ (x64)
- .NET 10 runtime
- The filesystem hosting VHDX files must be NTFS (no compression, no EFS)
- Administrator rights to install and start the service

## Getting Started

### 1. Build

```powershell
dotnet build vhdx-cow.slnx
```

### 2. Install the service

```powershell
# Publish a self-contained executable first
dotnet publish src/VhdxCow.Service/VhdxCow.Service.csproj -c Release -o publish/service

# Install (requires elevation)
.\scripts\Install-Service.ps1 -Action Install -ServicePath ".\publish\service\VhdxCow.Service.exe"
.\scripts\Install-Service.ps1 -Action Start
```

### 3. Configure allowed paths

Edit `%ProgramData%\VhdxCow\appsettings.json` (or the `appsettings.json` next to the service executable) to declare which directories the service will operate on:

```json
{
	"VhdxCow": {
		"AllowedParentPaths":    ["D:\\VhdxDisks\\Parents"],
		"AllowedMountBasePaths": ["D:\\GitHub"],
		"AllowedChildBasePaths": ["D:\\VhdxDisks\\Children"]
	}
}
```

The service rejects any request whose paths fall outside these allow-lists.

### 4. Use the CLI

```powershell
# Check connectivity
vhdx-cow ping

# Mount a worktree's bin/ from a parent VHDX
vhdx-cow init --parent D:\VhdxDisks\Parents\main.vhdx `
              --child  D:\VhdxDisks\Children\wt1.vhdx `
              --mount  D:\GitHub\myrepo-wt1\bin

# List all active mounts
vhdx-cow list

# Discard worktree changes — restore to parent state
vhdx-cow reset --child D:\VhdxDisks\Children\wt1.vhdx

# Show status of a specific mount
vhdx-cow status --child D:\VhdxDisks\Children\wt1.vhdx

# Detach and delete a worktree disk
vhdx-cow cleanup --child D:\VhdxDisks\Children\wt1.vhdx

# Merge main overlay into parent and refresh all children
vhdx-cow publish --overlay D:\VhdxDisks\main-overlay.vhdx
```

#### Global options

| Option | Default | Description |
|--------|---------|-------------|
| `--pipe-name` | `VhdxCowService` | Named pipe to connect to (useful when running multiple service instances) |
| `--timeout` | none | Operation timeout in seconds |

#### Exit codes

| Code | Meaning |
|------|---------|
| 0 | Success |
| 1 | Operational failure (see stderr for details) |
| 2 | Service unreachable (not running) |
| 3 | Timeout |

## Typical Workflow

```
                   ┌─────────────┐
                   │  main build │  → artifacts written to main-overlay.vhdx
                   └──────┬──────┘
                           │  vhdx-cow publish --overlay main-overlay.vhdx
                           ▼
                   parent VHDX updated
                   all children recreated
                           │
          ┌────────────────┼────────────────┐
          ▼                ▼                ▼
    worktree 1       worktree 2       worktree N
    (reset to        (reset to        (reset to
     parent)          parent)          parent)
          │
          │  build only changed projects
          ▼
    child VHDX stores only the diff
```

## Development

### Solution structure

```
vhdx-cow/
├── src/
│   ├── VhdxCow.Contracts/          # Protobuf definitions, ServiceConstants
│   │   └── Protos/vhdx_service.proto
│   ├── VhdxCow.Service/            # Windows Service
│   │   ├── Native/                 # CsWin32 function list and config
│   │   ├── Properties/             # appsettings.json
│   │   ├── Security/               # PathValidator
│   │   ├── Services/               # VhdxGrpcService (gRPC handlers)
│   │   ├── State/                  # JsonStateStore
│   │   └── VhdxOperations/         # VhdxManager (P/Invoke), VolumeManager
│   ├── VhdxCow.Client/             # IVhdxCowClient, VhdxCowClient, NamedPipeChannelFactory
│   └── VhdxCow.Cli/                # CommandFactory, Program.cs
├── tests/
│   ├── VhdxCow.Service.Tests/      # Unit tests (NUnit, NSubstitute)
│   ├── VhdxCow.Client.Tests/       # Client unit tests
│   ├── VhdxCow.Cli.Tests/          # CLI command tests with mock client
│   └── VhdxCow.Integration.Tests/  # Integration tests (require admin + VHDX)
└── scripts/
    └── Install-Service.ps1
```

### Running tests

```powershell
# Unit tests only (no admin required)
dotnet test vhdx-cow.slnx

# Integration tests (requires admin elevation and existing VHDX files)
dotnet test vhdx-cow.slnx --filter "Category=Integration"
```

### Service management

```powershell
.\scripts\Install-Service.ps1 -Action Status
.\scripts\Install-Service.ps1 -Action Stop
.\scripts\Install-Service.ps1 -Action Start
.\scripts\Install-Service.ps1 -Action Uninstall
```

### Logging

- **File**: `%ProgramData%\VhdxCow\logs\service-YYYYMMDD.log` (30-day rolling)
- **Event Log**: Application log, source `VhdxCow` (warnings and above)
- **Console**: Active when running interactively (not as a service)
