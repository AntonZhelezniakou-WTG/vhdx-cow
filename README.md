# vhdx-cow

VHDX-backed Copy-on-Write for git worktrees on Windows.

Build artifacts (e.g. `bin/`) in the main repository live on a parent VHDX disk. Each git worktree gets a _differencing_ (child) VHDX that stores only its own changes ‚Äî unmodified blocks are read transparently from the immutable parent. Resetting a worktree to the parent state takes milliseconds: the child disk is deleted and recreated from scratch.

Mounting a VHDX to an NTFS folder requires administrator privileges. A Windows Service handles all privileged disk operations; a user-space CLI communicates with it over a local gRPC/named-pipe channel.

## Architecture

```
main repo
‚îî‚îÄ‚îÄ bin/  ‚Üê‚îÄ‚îÄ main-overlay.vhdx  (differencing)
                      ‚îÇ
                      ‚ñº
              main.vhdx  (parent, READ-ONLY)
                      ‚îÇ
          ‚îå‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚î¥‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îê
          ‚ñº                       ‚ñº
  wt1-child.vhdx          wt2-child.vhdx
  worktree1/bin/          worktree2/bin/
```

The parent VHDX is **immutable**. Both the main repository overlay and each worktree child reference it as a read-only base. Modifying the parent (via the `publish` command) detaches all children, merges the overlay into the parent, and recreates all children from the new baseline.

### Components

| Project | Role |
|---------|------|
| `VhdxCow.Contracts` | Protobuf service definition, shared constants (`ServiceConstants`) |
| `VhdxCow.Service` | Windows Service ‚Äî gRPC server over named pipe, all privileged VHDX operations |
| `VhdxCow.Client` | Client library ‚Äî typed async wrapper over the gRPC channel |
| `VhdxCow.Cli` | Command-line tool for end users |

### IPC

The CLI connects to the service via a Windows named pipe (`\\.\pipe\VhdxCowService`) using gRPC/HTTP2. Named pipe ACLs grant access to `BUILTIN\Users`; `TokenImpersonationLevel.Anonymous` prevents privilege escalation from the client side.

### VHDX operations

The service calls the Win32 VirtDisk API directly via P/Invoke (generated by [CsWin32](https://github.com/microsoft/CsWin32)) ‚Äî no dependency on Hyper-V PowerShell cmdlets or the Hyper-V role.

Key API calls: `CreateVirtualDisk` ¬∑ `OpenVirtualDisk` ¬∑ `AttachVirtualDisk` ¬∑ `DetachVirtualDisk` ¬∑ `GetVirtualDiskPhysicalPath` ¬∑ `MergeVirtualDisk` ¬∑ `SetVolumeMountPoint` ¬∑ `DeleteVolumeMountPoint`

Attached disks use the `PERMANENT_LIFETIME` flag ‚Äî mounts survive service restarts. Active mounts are tracked in `%ProgramData%\VhdxCow\state.json` and reconciled on startup.

## Requirements

- Windows 10/11 or Windows Server 2019+ (x64)
- .NET 10 runtime
- The filesystem hosting VHDX files must be NTFS (no compression, no EFS)
- Administrator rights to install and start the service

## Installation

### MSI installer (recommended)

The MSI installer handles everything: service registration, CLI placement, PATH configuration, upgrade, and uninstall.

```powershell
# Silent install (requires elevation)
msiexec /i VhdxCow-1.0.0.msi /qn

# Interactive install
msiexec /i VhdxCow-1.0.0.msi
```

After installation:
- Service `VhdxCowService` is registered and started automatically (auto-start, LocalSystem)
- CLI is placed in `C:\Program Files\VhdxCow\Cli\` and added to the system PATH
- Default config is created at `%ProgramData%\VhdxCow\appsettings.json`

**Upgrade**: run the new MSI ‚Äî the old version is stopped and removed first, then the new one is installed.

**Uninstall**: removes the service and CLI from PATH; `%ProgramData%\VhdxCow\` (config and logs) is preserved.

```powershell
msiexec /x VhdxCow-1.0.0.msi /qn
```

## Getting Started

### 1. Build

```powershell
dotnet build vhdx-cow.slnx
```

### 2. Install the service

#### Via MSI (see [Installation](#installation) above)

The MSI is built separately from the main solution ‚Äî see [Building the MSI](#building-the-msi).

#### Manually (for development)

```powershell
# Publish first
dotnet publish src/VhdxCow.Service/VhdxCow.Service.csproj -c Release -o publish/service

# Install (requires elevation)
.\scripts\Install-Service.ps1 -Action Install -ServicePath ".\publish\service\VhdxCow.Service.exe"
.\scripts\Install-Service.ps1 -Action Start
```

### 3. Configure allowed paths

Edit `%ProgramData%\VhdxCow\appsettings.json` (or the `appsettings.json` next to the service executable) to declare which directories the service will operate on:

```json
{
	"VhdxCow": {
		"AllowedParentPaths":    ["D:\\VhdxDisks\\Parents"],
		"AllowedMountBasePaths": ["D:\\GitHub"],
		"AllowedChildBasePaths": ["D:\\VhdxDisks\\Children"]
	}
}
```

The service rejects any request whose paths fall outside these allow-lists.

### 4. Use the CLI

```powershell
# Check connectivity
vhdx-cow ping

# Mount a worktree's bin/ from a parent VHDX
vhdx-cow init --parent D:\VhdxDisks\Parents\main.vhdx `
              --child  D:\VhdxDisks\Children\wt1.vhdx `
              --mount  D:\GitHub\myrepo-wt1\bin

# List all active mounts
vhdx-cow list

# Discard worktree changes ‚Äî restore to parent state
vhdx-cow reset --child D:\VhdxDisks\Children\wt1.vhdx

# Show status of a specific mount
vhdx-cow status --child D:\VhdxDisks\Children\wt1.vhdx

# Detach and delete a worktree disk
vhdx-cow cleanup --child D:\VhdxDisks\Children\wt1.vhdx

# Merge main overlay into parent and refresh all children
vhdx-cow publish --overlay D:\VhdxDisks\main-overlay.vhdx
```

#### Global options

| Option | Default | Description |
|--------|---------|-------------|
| `--pipe-name` | `VhdxCowService` | Named pipe to connect to (useful when running multiple service instances) |
| `--timeout` | none | Operation timeout in seconds |

#### Exit codes

| Code | Meaning |
|------|---------|
| 0 | Success |
| 1 | Operational failure (see stderr for details) |
| 2 | Service unreachable (not running) |
| 3 | Timeout |

### Building the MSI

The installer project (`installer/VhdxCow.Installer.wixproj`) is **not part of `vhdx-cow.slnx`** ‚Äî it must be built separately from the command line. This is because `.wixproj` files require the [HeatWave](https://marketplace.visualstudio.com/items?itemName=FireGiant.FireGiantHeatWaveDev17) Visual Studio extension to load in the IDE, and the WiX v5 SDK is not supported by all Visual Studio configurations.

```powershell
# 1. Publish both components
dotnet publish src/VhdxCow.Service -c Release --self-contained false -r win-x64 -o publish/service
dotnet publish src/VhdxCow.Cli    -c Release --self-contained false -r win-x64 -o publish/cli

# 2. Build the MSI
dotnet build installer/VhdxCow.Installer.wixproj -c Release -p:Version=1.0.0
# ‚Üí installer/bin/Release/VhdxCow-1.0.0.msi
```

The SDK package (`WixToolset.Sdk`) is declared in the `.wixproj` and restored automatically by `dotnet build` ‚Äî no separate WiX installation is required.

### WiX support in Visual Studio

The installer project uses `WixToolset.Sdk` which Visual Studio does not support out of the box. Without the extension, the project loads as `(incompatible)` in Solution Explorer but still builds correctly from the command line.

To get full IDE support (syntax highlighting, IntelliSense for `.wxs` files, build integration):

1. Open Visual Studio ‚Üí **Extensions** ‚Üí **Manage Extensions**
2. Search for **HeatWave**
3. Install **HeatWave for VS 2022** (works in VS 2022 and later)
4. Restart Visual Studio

Direct link: `https://marketplace.visualstudio.com/items?itemName=FireGiant.FireGiantHeatWaveDev17`

After installation the project loads normally and the `(incompatible)` label disappears.
