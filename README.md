# VHDX Manager

VHDX management tool for Windows — create, mount, and merge virtual disks from a single CLI.

Useful for any workflow that benefits from a private NTFS or ReFS volume backed by a `.vhdx` file: isolated build outputs, ephemeral scratch spaces, portable dependency caches, copy-on-write snapshots, and similar. The tool itself is agnostic — it just manages VHDXes.

Mounting a VHDX to an NTFS folder requires administrator privileges. A Windows Service handles all privileged disk operations; a user-space CLI (`vhmgr`) communicates with it over a local gRPC/named-pipe channel.

---

## Architecture

```
overlay.vhdx  (differencing, mutable)
      │
      ▼
parent.vhdx  (base image, READ-ONLY)
      │
 ┌────┴────┐
 ▼         ▼
child-A.vhdx   child-B.vhdx
mount-A/       mount-B/
```

The parent VHDX is **immutable**. Each child references it as a read-only base layer and stores only the delta. Modifying the parent (via the `publish` command) detaches all children, merges the overlay into the parent, and recreates all children from the new baseline.

### Components

| Project | Role |
|---------|------|
| `VhdxManager.Contracts` | Protobuf service definition, shared constants (`ServiceConstants`) |
| `VhdxManager.Service` | Windows Service — gRPC server over named pipe, all privileged VHDX operations |
| `VhdxManager.Client` | Client library — typed async wrapper over the gRPC channel |
| `VhdxManager.Cli` | Command-line tool for end users |

### IPC

The CLI connects to the service via a Windows named pipe (`\\.\pipe\VhdxManagerService`) using gRPC/HTTP2. Named pipe ACLs grant access to `BUILTIN\Users`; `TokenImpersonationLevel.Anonymous` prevents privilege escalation from the client side.

### VHDX operations

The service calls the Win32 VirtDisk API directly via P/Invoke (generated by [CsWin32](https://github.com/microsoft/CsWin32)) — no dependency on Hyper-V PowerShell cmdlets or the Hyper-V role.

Key API calls: `CreateVirtualDisk` · `OpenVirtualDisk` · `AttachVirtualDisk` · `DetachVirtualDisk` · `GetVirtualDiskPhysicalPath` · `MergeVirtualDisk` · `SetVolumeMountPoint` · `DeleteVolumeMountPoint`

Attached disks use the `PERMANENT_LIFETIME` flag — mounts survive service restarts. Active mounts are tracked in `%ProgramData%\VhdxManager\state.json` and reconciled on startup.

## Requirements

- Windows 10/11 or Windows Server 2019+ (x64)
- .NET 10 runtime
- The filesystem hosting VHDX files must be NTFS (no compression, no EFS)
- Administrator rights to install and start the service

## Installation

### MSI installer (recommended)

The MSI installer handles everything: service registration, CLI placement, PATH configuration, upgrade, and uninstall.

```powershell
# Silent install (requires elevation)
msiexec /i VhdxManager-1.0.0.msi /qn

# Interactive install
msiexec /i VhdxManager-1.0.0.msi
```

After installation:
- Service `VhdxManagerService` is registered and started automatically (auto-start, LocalSystem)
- CLI is placed in `C:\Program Files\VhdxManager\Cli\` and added to the system PATH
- Default config is created at `%ProgramData%\VhdxManager\appsettings.json`

**Upgrade**: run the new MSI — the old version is stopped and removed first, then the new one is installed.

**Uninstall**: removes the service and CLI from PATH; `%ProgramData%\VhdxManager\` (config and logs) is preserved.

```powershell
msiexec /x VhdxManager-1.0.0.msi /qn
```

## Getting Started

### 1. Build

```powershell
dotnet build VhdxManager.slnx
```

### 2. Install the service

#### Via MSI (see [Installation](#installation) above)

The MSI is built separately from the main solution — see [Building the MSI](#building-the-msi).

#### Manually (for development)

```powershell
# Publish first
dotnet publish src/VhdxManager.Service/VhdxManager.Service.csproj -c Release -o publish/service

# Install (requires elevation)
.\scripts\Install-Service.ps1 -Action Install -ServicePath ".\publish\service\VhdxManager.Service.exe"
.\scripts\Install-Service.ps1 -Action Start
```

### 3. Configure allowed paths

Edit `%ProgramData%\VhdxManager\appsettings.json` (or the `appsettings.json` next to the service executable) to declare which directories the service will operate on. By default all four lists contain `"*"` (allow everything) — replace with concrete paths to restrict.

```json
{
	"VhdxManager": {
		"AllowedParentPaths":    ["D:\\VhdxDisks\\Parents"],
		"AllowedMountBasePaths": ["D:\\MountPoints"],
		"AllowedChildBasePaths": ["D:\\VhdxDisks\\Children"]
	}
}
```

The service rejects any request whose paths fall outside these allow-lists.

### 4. Use the CLI

```powershell
# Check connectivity
vhmgr ping

# Create a standalone VHDX, format it, and mount it to a folder
vhmgr create --path D:\VhdxDisks\data.vhdx --size 50G --label data `
             --mount D:\MountPoints\data

# Create a differencing child off a parent VHDX and mount it
vhmgr init --parent D:\VhdxDisks\Parents\base.vhdx `
           --child  D:\VhdxDisks\Children\snap1.vhdx `
           --mount  D:\MountPoints\snap1

# `create --parent` is equivalent to `init`
vhmgr create --parent D:\VhdxDisks\Parents\base.vhdx `
             --path   D:\VhdxDisks\Children\snap2.vhdx `
             --mount  D:\MountPoints\snap2

# List all active mounts
vhmgr list

# Show status of a specific mounted disk
vhmgr status --child D:\VhdxDisks\Children\snap1.vhdx

# Discard all changes in a child disk — restore to parent state
vhmgr reset --child D:\VhdxDisks\Children\snap1.vhdx

# Detach and delete a child disk
vhmgr cleanup --child D:\VhdxDisks\Children\snap1.vhdx

# Merge an overlay into its parent and recreate all children from the new baseline
vhmgr publish --overlay D:\VhdxDisks\overlay.vhdx

# Attach + mount an existing VHDX without creating it
vhmgr mount --path D:\VhdxDisks\data.vhdx --mount D:\MountPoints\data

# Unmount + detach, keeping the file on disk
vhmgr unmount --path D:\VhdxDisks\data.vhdx

# Unmount + detach + delete the file
vhmgr delete D:\VhdxDisks\data.vhdx

# Convert an existing folder into a VHDX-backed folder
vhmgr convert --folder D:\Data --vhdx D:\VhdxDisks\data.vhdx --size 100G
```

#### Global options

| Option | Default | Description |
|--------|---------|-------------|
| `--pipe-name` | `VhdxManagerService` | Named pipe to connect to (useful when running multiple service instances) |
| `--timeout` | none | Operation timeout in seconds |

#### Exit codes

| Code | Meaning |
|------|---------|
| 0 | Success |
| 1 | Operational failure (see stderr for details) |
| 2 | Service unreachable (not running) |
| 3 | Timeout |

## Differencing Disk Workflow

The `publish` command supports a copy-on-write pattern where multiple mounted children share a single read-only parent:

```
                ┌──────────────┐
                │  overlay.vhdx│  ← accumulate changes here
                └──────┬───────┘
                       │  vhmgr publish --overlay overlay.vhdx
                       ▼
               parent VHDX updated
               all children recreated
                       │
        ┌──────────────┼──────────────┐
        ▼              ▼              ▼
  child-A.vhdx   child-B.vhdx   child-C.vhdx
  mount-A/       mount-B/       mount-C/
  (reset to      (reset to      (reset to
   parent)        parent)        parent)
```

Each child stores only its own delta on top of the shared parent. `reset` discards a child's changes without touching the parent or other children.

### Building the MSI

The installer project (`installer/VhdxManager.Installer.wixproj`) is **not part of `VhdxManager.slnx`** — it must be built separately from the command line. This is because `.wixproj` files require the [HeatWave](https://marketplace.visualstudio.com/items?itemName=FireGiant.FireGiantHeatWaveDev17) Visual Studio extension to load in the IDE, and the WiX v5 SDK is not supported by all Visual Studio configurations.

```powershell
# 1. Publish both components
dotnet publish src/VhdxManager.Service -c Release --self-contained false -r win-x64 -o publish/service
dotnet publish src/VhdxManager.Cli    -c Release --self-contained false -r win-x64 -o publish/cli

# 2. Build the MSI
dotnet build installer/VhdxManager.Installer.wixproj -c Release -p:Version=1.0.0
# → installer/bin/Release/VhdxManager-1.0.0.msi
```

The SDK package (`WixToolset.Sdk`) is declared in the `.wixproj` and restored automatically by `dotnet build` — no separate WiX installation is required.

### Signing the MSI for development

By default Windows displays an "Unknown publisher" UAC prompt for an unsigned MSI. To replace this with `Verified publisher: WiseTechGlobal` + `VHDX Manager` on your dev box:

```powershell
# One-time per dev machine, requires admin elevation
.\scripts\Setup-DevCodeSigning.ps1
```

This generates `installer/dev-cert.pfx` (gitignored) and trusts the cert in `LocalMachine\Root` + `LocalMachine\TrustedPublisher`. Subsequent MSI builds detect the PFX automatically and call `signtool sign` from the post-build target in [installer/VhdxManager.Installer.wixproj](installer/VhdxManager.Installer.wixproj).

For production, override the properties when building:

```powershell
dotnet build installer/VhdxManager.Installer.wixproj -c Release `
    /p:DevCertPath=path\to\real-codesign.pfx `
    /p:DevCertPassword=$env:CODESIGN_PASSWORD
```

### Releases (GitHub Actions)

The [`.github/workflows/release.yml`](.github/workflows/release.yml) workflow builds, signs, and publishes a GitHub Release containing the signed MSI.

**To cut a release**: push a tag like `v1.0.0`. The workflow takes the version from the tag, builds + tests + publishes Service and CLI, builds the MSI, signs it if the signing secrets are present, and creates a GitHub Release with the MSI attached.

```powershell
git tag v1.0.0
git push origin v1.0.0
```

The release notes are auto-generated from PRs/commits since the previous tag.

**To dry-run** without publishing: trigger the workflow manually from the Actions tab with a version input. The result is a *draft* release at the same tag; review the MSI, then click **Publish release** when ready.

**Signing secrets** (optional — without them the MSI is built unsigned and Windows shows "Unknown publisher" on the UAC prompt):

| Secret | Contents |
|---|---|
| `CODESIGN_PFX_BASE64` | base64-encoded `.pfx` file (`[Convert]::ToBase64String([IO.File]::ReadAllBytes('cert.pfx'))`) |
| `CODESIGN_PFX_PASSWORD` | password for the `.pfx` |
