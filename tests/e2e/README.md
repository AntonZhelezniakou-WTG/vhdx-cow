# VhdxManager E2E test rig

End-to-end tests for the full installation + service + CLI surface, running
against a Hyper-V virtual machine.

## Why a VM

The service runs as `LocalSystem`, partitions disks via Win32 IOCTLs, calls
`Format-Volume` and `Add-MpPreference`, and is delivered through an MSI that
registers a Windows Service. None of this is faithfully testable inside a
unit-test harness on the developer box, and running it directly would leave
real system state behind every run. A throwaway VM is the only honest option.

## What's in this folder

```
tests/e2e/
├── Bootstrap-VM.ps1                 # one-shot host-side script (run once)
├── lib/
│   └── Helpers.ps1                  # shared helpers (ISO authoring, PS Direct, ...)
├── templates/
│   ├── autounattend.xml.template    # Windows Setup answer file
│   └── FirstLogon.ps1               # guest-side first-boot config
├── .vm-creds.json                   # generated by Bootstrap-VM.ps1, gitignored
└── README.md                        # this file
```

## Why Generation 1 (BIOS) and not Generation 2 (UEFI)

Hyper-V Gen2 + UEFI Secure Boot rejects the bootloader on some Windows ISOs
(notably LTSC Eval after Microsoft's August 2024 DBX update revoked older
Windows bootloaders for the BlackLotus mitigation), producing
`"The boot loader failed"` or `"The signed image's hash is not allowed (DB)"`
at first boot. Disabling Secure Boot on Gen2 doesn't fully fix it because
the bootloaders themselves expect a signed boot environment.

Gen1 is BIOS-only — there's no signature check at all, so any ISO that
happens to be UEFI-only-troublesome boots cleanly. The trade-off (no vTPM,
no Secure Boot inside the VM) is irrelevant to the MSI under test, and Win11
Setup's hardware checks for those are bypassed via registry tweaks injected
during WinPE (see `templates/autounattend.xml.template`).

## Host prerequisites

* Windows 10/11 Pro/Enterprise or Server 2019+ with **Hyper-V enabled**.
* **PowerShell 5.1 or newer** — 5.1 is the default on Win 10 (1607+) / Win 11
  / Server 2016+, so you almost certainly already have it. PowerShell 7 is
  *not* required (and not forced); the script checks the version on startup
  and prints a download link if your host is somehow older.
* ~50 GB free disk space for the VM and ISO.

The script **self-elevates** if you launch it from a non-admin shell —
you'll see a UAC prompt and the script then continues in a new elevated
window. Click "Yes" on the UAC dialog to proceed.

To check Hyper-V is on:

```powershell
Get-WindowsOptionalFeature -Online -FeatureName Microsoft-Hyper-V-All
```

To enable it (one-time, requires reboot):

```powershell
Enable-WindowsOptionalFeature -Online -FeatureName Microsoft-Hyper-V-All -All
```

## One-shot bootstrap

From any PowerShell window (no need to right-click → Run as Administrator —
the script will request elevation itself):

```powershell
cd D:\path\to\vhdx-cow\tests\e2e
.\Bootstrap-VM.ps1
```

A file picker opens — point it at any Win11 Eval ISO (regular Enterprise or
LTSC, both work). If you don't have one yet, cancel the dialog: the script
prints the Microsoft Evaluation Center URL, optionally opens it in your
browser, and exits.

The script will:

1. Mount the ISO briefly to detect which Windows edition is inside.
   For single-image ISOs (typical Eval) it picks that automatically; for
   multi-image ISOs it auto-selects against a list of common Enterprise/
   Eval SKUs and only prompts if nothing matches.
2. Pick `C:\Hyper-V` as the VM root if it exists, otherwise pop a
   folder-picker dialog. The VM ends up in `<root>\VhdxManagerE2E\`.
3. If a VM with the same name (or stale files in the VM directory) already
   exists, ask before deleting. Pass `-Force` to skip the prompt.
4. Generate a random local-admin password, save it to `.vm-creds.json`.
5. Pack `autounattend.xml` + `FirstLogon.ps1` into `autounattend.iso`.
6. Create a **Generation 1** VM (2 vCPU / 4 GB dynamic / 32 GB OS VHDX /
   **no virtual NIC**). The Win11 ISO + autounattend ISO are attached as
   IDE DVD drives; BIOS startup order is set so the install ISO boots first.
7. Boot it. The VM runs Windows Setup unattended, with TPM/Secure Boot/CPU/
   RAM/storage hardware checks bypassed via WinPE registry tweaks. Wait
   until the guest writes `C:\Setup\boot-complete.flag` (typically
   8–15 minutes for the unattended install + first logon).
8. Shut down cleanly, detach the install ISO, take the
   `pre-install-clean` checkpoint.

To wipe and redo from scratch without the confirmation prompt:

```powershell
.\Bootstrap-VM.ps1 -Force
```

## Silent / scripted mode

For CI and other unattended invocations, every input the script would
normally prompt for can be passed on the command line, and the `-Silent`
switch disables every interactive prompt / dialog / browser popup.

```powershell
# Must be run from an already-elevated PowerShell — silent mode refuses
# to trigger UAC because UAC requires interaction.
.\Bootstrap-VM.ps1 `
    -Silent `
    -IsoPath  'D:\ISOs\Win11_24H2_Eval.iso' `
    -VmRoot   'D:\TestVMs' `
    -Force
```

| Parameter | Purpose | Required in `-Silent`? |
|---|---|---|
| `-IsoPath` | Path to the Win11 Eval ISO | **Yes** |
| `-VmRoot` | Parent dir; VM stored in `<VmRoot>\VhdxManagerE2E\` | Only if `C:\Hyper-V` doesn't exist |
| `-ImageName` | Override the WIM image name. Without it the script auto-picks (single-image ISO → that one; multi-image → first match against common Eval/Enterprise SKUs). | No |
| `-VmName` | Hyper-V VM display name (default `VhdxManagerE2E`) | No |
| `-Force` | Skip the "Remove existing VM?" confirmation. Required in `-Silent` if the VM already exists. | Only if VM exists |
| `-Silent` | Disable all prompts; fail on missing required inputs | n/a |

In silent mode the script:

* Refuses to self-elevate (UAC needs a user) — invoke from an already-admin shell.
* Validates `-IsoPath` exists and ends in `.iso` — throws otherwise.
* Skips the file picker / "open browser?" prompts.
* Skips the folder-picker dialog for the VM root.

Exit codes:

* `0` — bootstrap succeeded.
* non-zero — clear error message on stderr (missing required param, ISO
  not found, Hyper-V not installed, UAC declined, guest first-boot
  timed out, …).

## Resulting state

After bootstrap completes:

| Artefact | Where |
|---|---|
| VM | `<root>\VhdxManagerE2E\` (visible in Hyper-V Manager) |
| Generation | 1 (BIOS, no vTPM, no Secure Boot) |
| OS disk | `<root>\VhdxManagerE2E\os.vhdx` |
| autounattend.iso | `<root>\VhdxManagerE2E\autounattend.iso` (kept for diagnostics) |
| Checkpoint | `pre-install-clean` |
| Guest user | `vhdxtest` (local admin), password in `.vm-creds.json` |
| Guest networking | none (PowerShell Direct via VMBus) |
| Windows Update | disabled on guest |
| Sleep / hibernate | disabled on guest |

## Manual inspection

Log into the VM by hand if a test fails and you want to poke around:

```powershell
Start-VM -Name VhdxManagerE2E
vmconnect.exe localhost VhdxManagerE2E
# Login as vhdxtest with the password from .vm-creds.json
```

Reset to clean state:

```powershell
Restore-VMSnapshot -VMName VhdxManagerE2E -Name pre-install-clean -Confirm:$false
```

## How tests will use this VM (next phase)

* Per fixture, restore a checkpoint (either `pre-install-clean` or
  `installed-clean` — the latter created by the test runner the first time
  it sees a new MSI).
* Tests run as `Invoke-Command -VMName VhdxManagerE2E -Credential ...`
  blocks that exec `vhmgr.exe` and assert state via the guest's filesystem
  / `Get-Service` / `Get-MpPreference`.
* No host↔guest networking → MSI is shipped over `Copy-Item -ToSession
  (New-PSSession -VMName ...)`.

The test project itself (`tests/VhdxManager.E2E.Tests`) will be added in
the next step. This folder produces only the VM the tests run against.

## Why no virtual NIC

Without a network adapter the guest can't:

* check for Windows updates,
* phone home to telemetry endpoints,
* download additional Defender signatures mid-test,

…all of which would otherwise cause non-deterministic delays and CPU spikes
in tests. Communication to the guest is exclusively via `Invoke-Command
-VMName`, which uses the Hyper-V VMBus and the guest's local WinRM listener,
not any TCP path. Add a `-SwitchName` to the `New-VM` line in
`Bootstrap-VM.ps1` if you ever need network access for debugging.

## Troubleshooting

**"Wait-VmReady: Guest did not become ready within 45 minutes"**
Most likely the WIM image name in `autounattend.xml` doesn't match your
ISO. Mount the ISO and run:

```powershell
Dism /Get-WimInfo /WimFile:E:\sources\install.wim
```

Pass the correct name via `-ImageName` and re-run with `-Force`:

```powershell
# Example for LTSC Eval ISO:
.\Bootstrap-VM.ps1 -ImageName 'Windows 11 Enterprise LTSC Evaluation' -Force
```

Also worth checking via vmconnect: did Setup actually start? If the BIOS
shows "no bootable device", the install ISO didn't get tried first — verify
`Set-VMBios -StartupOrder` ran with `CD` in the first position.

**"Elevation failed (UAC declined or denied by policy)"**
You clicked "No" on the UAC prompt, or your machine policy disallows UAC
elevation for the current account. Re-run the script and accept the
prompt, or right-click PowerShell → "Run as administrator" first to
bypass the self-elevation step.

**"PowerShell 5.1 or newer is required (current: ...)"**
You're on Windows 7 / 8.1 or have a non-default PowerShell. Install the
latest PowerShell 7 from the GitHub releases page printed by the script
and re-run.

**Wrong password / can't log in via vmconnect**
Open `.vm-creds.json` and copy the password verbatim. The auto-logon
expires after 3 logons; thereafter you must enter it manually.

**Setup gets stuck on "This PC can't run Windows 11"**
The hardware-check bypass keys in `autounattend.xml` weren't applied —
either the autounattend.iso wasn't picked up by Setup, or a newer Win11
build added a new check we don't bypass. Open vmconnect during install:
if you see a green "Setup" window with a hardware-compatibility error,
the answer file isn't being read. The DVD letter scan in the WinPE
RunSynchronous block (`for %i in (D E F G H) do ...`) may need extending
if your host has unusual drive-letter assignments inside WinPE.
